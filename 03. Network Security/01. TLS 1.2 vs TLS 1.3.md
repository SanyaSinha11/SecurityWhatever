# Overview

TLS 1.3 is significantly faster and more secure than TLS 1.2. It reduces connection latency by halving the handshake process to a single round trip, removes vulnerable legacy algorithms, and encrypts more of the handshake data to better protect user privacy.

# Key Differences at a Glance

| Feature                        | TLS 1.2                                                                                                             | TLS 1.3                                                                                            |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Handshake Speed                | 2 Round Trips (2-RTT) required to establish a secure connection.                                                    | 1 Round Trips (1-RTT) required - cuts connection setup time in half.                               |
| Session Resumption             | 1 Round Trip (1-RTT)                                                                                                | 0 Round Trip (0-RTT), repeat visitors can send encrypted data immediately without a handshake.     |
| Cipher Suites                  | 200+ combinations; Supports both strong and weak cipher suites; complex configurations are required to stay secure. | 5 secure options; Weak ciphers removed; strictly enforces secure, modern cryptographic algorithms. |
| Forward Secrecy                | Optional                                                                                                            | Mandatory                                                                                          |
| Handshake Privacy (Encryption) | Plaintext elements; Many negotiation elements are sent in plaintext.                                                | Fully encrypted; Most of the handshake is encrypted, hiding sensitive server certificate details.  |
| Insecure Algorithms            | Allowed (MD5, SHA-1, RC4)                                                                                           | Completely removed                                                                                 |

# Attacks and Bypass
### Why TLS 1.2 is considered less secure?
The primary insecurity in TLS 1.2 stems from "cipher sprawl." Because the protocol was designed to be highly flexible with older systems, it historically allowed servers and clients to negotiate weakened, primitive, or export-grade cryptography.

## Attacks and Bypass in TLS 1.2
#### **1. Downgrade Attacks**

- **The Problem:** Attackers perform a Man-in-the-Middle (MitM) intercept between a client and server, modifying the list of supported ciphers or protocol versions to force them to fall back to older, vulnerable protocols (like SSL 3.0).
  
- **Examples:** The FREAK and Logjam attacks manipulated the protocol to force the use of weaker "export-grade" 512-bit encryption keys. 

#### **2. Block Cipher Vulnerabilities (CBC Mode)**

- **The Problem:** Cipher Block Chaining (CBC) modes used in TLS 1.2 have known side-channel weaknesses where the ciphertext leaks information about the plaintext padding. 
  
- **Examples:** The POODLE (Padding Oracle On Downgraded Legacy Encryption) and Lucky13 attacks exploit these timing differences and padding oracles to decrypt sensitive data like authentication cookies byte by byte.

#### **3. Key Exchange Weaknesses (Static RSA)**

- **The Problem:** If TLS 1.2 uses static RSA for key exchange, anyone who obtains the server's long-term private key can decrypt all past intercepted communications, as it is used directly to derive the session keys.
  
- **Attack:** Attacks like ROBOT exploit weaknesses in RSA encryption padding oracles. 

#### **4. Compression Side-Channel Attacks**

- **The Problem:** TLS supports compression. If an attacker can inject text into the victim's request, they can guess what is being compressed by observing the length of the resulting encrypted payload.
  
- **Examples:** The CRIME and BREACH attacks allow hackers to reveal hidden plaintext, such as CSRF tokens.

#### NOTE:
TLS 1.2 is vulnerable to a specific variant of the **POODLE attack**, but there is a crucial distinction between why it affects SSL 3.0 versus TLS 1.2
- **In SSL 3.0**, POODLE (CVE-2014-3566) is a **protocol flaw**. The SSL 3.0 specification inherently does not define what the padding bytes should look like, meaning every single implementation of SSL 3.0 is fundamentally broken. 
  
- **In TLS 1.2**, POODLE is an **implementation flaw** (such as CVE-2014-8730). The TLS 1.2 specification strictly dictates how padding must be formatted and checked. However, certain software developers took shortcuts when writing their TLS 1.2 code, failing to properly validate the padding blocks. 

## Attacks and Bypass in TLS 1.3 
TLS 1.3 has a vastly reduced attack surface. It removes support for static RSA, static Diffie-Hellman, RC4, DES, and all non-AEAD (Authenticated Encryption with Associated Data) ciphers. Because of these strict mandates, the historical attacks mentioned above do not work on TLS 1.3. However, it does face specialized edge-case risks:

#### **1. 0-RTT (Zero Round Trip Time) Replay Attacks**

- **The Problem:** TLS 1.3 offers a performance feature called 0-RTT, which allows the client to send encrypted data on the very first message. 
  
- **The Risk:** If the connection is intercepted, an attacker could potentially "replay" (resend) this initial 0-RTT request multiple times. Unless the application server implements strict anti-replay protections, this could allow malicious duplication of actions (like submitting multiple purchases).

#### **2. Downgrade Protection Spoofing**

- **The Problem:** While TLS 1.3 includes built-in downgrade protection signals, older, non-compliant firewalls, load balancers, or proxies have sometimes been tricked by attackers into removing these signals, causing connections to drop to TLS 1.2.

## How to Mitigate Risks in TLS 1.2 and TLS 1.3?
To prevent these exploits in legacy TLS 1.2 implementations, administrators must enforce strict configurations.

- **Disable Legacy Features:** Turn off support for SSLv3, TLS 1.0, TLS 1.1, and disable CBC-mode ciphers, static RSA, and RC4.
  
- **Enforce Perfect Forward Secrecy:** Only allow Ephemeral Diffie-Hellman (DHE) or Elliptic Curve (ECDHE) key exchanges.
  
- **Adopt AEAD Ciphers:** Force the use of authenticated encryption like AES-GCM or ChaCha20-Poly1305.
  
- **Upgrade to TLS 1.3:** Configure servers to prioritize TLS 1.3 for all modern clients to get built-in protection against algorithm downgrades.
