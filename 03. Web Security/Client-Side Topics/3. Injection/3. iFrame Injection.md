## Overview

An iFrame injection occurs when a malicious actor embeds a hidden or deceptive inline frame (`<iframe>`) into a legitimate website. This allows attackers to secretly load external, malicious webpages, steal user credentials, or trick users into performing unwanted actions without their knowledge.

An `<iframe>` is just a way to embed another webpage inside the current page. If there is any input that is shown back on the page, that means the input isn’t sanitized and is being reflected on the page.

- **XSS** = you can run code
- **Iframe injection** = you can inject malicious content/UI

## How it works

Attackers typically inject these iframes by exploiting website vulnerabilities like Cross-Site Scripting (XSS) or SQL Injection. When a user visits the compromised website, the browser automatically loads the embedded iframe in the background.

Common methods and goals of iFrame injection include:
- **Phishing & Fake Login Forms:** Attackers inject a fake login screen (for a bank, email, or social media platform) within a transparent or invisible iframe. When the user thinks they are logging into the legitimate site, they are actually submitting their credentials to the attacker's server.
  
- **Clickjacking:** The attacker layers a tiny or transparent iframe over legitimate buttons on the website. When the user attempts to click a real button on the page, they are tricked into clicking the hidden malicious element, which could authorize transactions or download malware.
  
- **Drive-by Downloads:** The malicious iframe secretly directs the user's browser to an "exploit kit" site that actively scans the visitor's machine for software vulnerabilities.

## Root Cause 

Server-side validation, as client-side validation only helps to - 
- Prevent accidental bad input
- Improve UX
- Reduce junk requests
- Give instant feedback
  
## Fixes

- Always **HTML-escape** user input before rendering
- Use templating engines that auto-escape
- Apply **Content Security Policy (CSP) - frame-src 'self'; frame-ancestors 'none';**
- Validate input (length, characters, format)
- Never directly insert raw user input into HTML
- Protect against reflected input issues
- On server-side: Reject or normalize anything outside: `[a-zA-Z0-9\s]`; Trim length; Drop control characters
- Don’t show raw, show processed and safer version
- Handle redirects safely - use an allow list of internal paths, ignore bad values, never redirect to user provided URLs
