## Cross-Site Scripting (XSS) Unscripted

> “From Testing Inputs to Thinking Like an Attacker, When Text Turns Toxic!”

Good security decisions come from good data, that’s what I was always. But raw logs alone aren’t enough — you need to connect and interpret them so they tell a clear story. That’s where the **three C’s** help:

- **Context:** understand what the data is about (what system, what user, what asset, what “normal” looks like).
- **Correlation:** connect related signals across tools and time (multiple small alerts that together point to one incident).
- **Causation:** figure out what actually caused the event and what it led to (root cause and impact), so you can fix the real problem — not just the symptom.

---
# Overview
XSS happens when an application takes user input and renders it in a browser without proper handling.
Instead of:
```
User input → displayed as text
```
Browser interprets it as active content.

# Types of XSS
### 1. Reflected XSS
Input is immediately reflected in the response.
Like alert pop ups, or search query results.

### 2. Stored XSS
Input gets saved.
Like: Comment fields, usernames, etc.
User submits content.
Anyone viewing the page later triggers the issue.

### 3. DOM XSS
JavaScript in the browser processes unsafe data.
Input may never reach the server.

Source:
```
URL
Fragment
window.location
document.URL
```

Sink:
```
innerHTML
document.write
```
