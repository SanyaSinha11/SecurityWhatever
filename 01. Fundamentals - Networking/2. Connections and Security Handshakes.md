Think of a website connection as setting up a secure, private meeting. Before you can exchange secrets, you first have to make sure the other person can hear you (the **TCP Handshake**), and then prove who you are and lock the door (the **TLS Handshake**)

# TCP Handshake
-***Establishing the Connection***

The TCP Handshake is a 3-way conversation that confirms both your browser (Client) and the server are online and ready to transmit data. It works like a walkie-talkie check:
```
[ Client ] --- 1. SYN (Can you hear me?) ---> [ Server ] 
[ Client ] <-- 2. SYN-ACK (Yes! Can you?) --- [ Server ] 
[ Client ] --- 3. ACK (Yes, loud & clear!) -> [ Server ]
```

1. **SYN (Synchronize):** Your browser sends a packet saying, _"Hello, I want to talk to you. Here is my starting sequence number."_
2. **SYN-ACK (Synchronize-Acknowledge):** The server replies, _"I received your message! I'm ready too. Here is my sequence number."_
3. **ACK (Acknowledge):** Your browser responds, _"Got it! Let's start talking."_ 

Now that the connection line is open, they need to secure it using **HTTPS**.

# TLS Handshake
-***Securing the Connection***

The TLS Handshake happens right after TCP. It determines which version of encryption to use, verifies the server's identity, and creates unique keys to lock the data.

- **Client Hello:** Your browser sends a greeting listing its supported TLS versions and cryptographic algorithms (Cipher Suites). 
- **Server Hello & Certificate:** The server picks the best encryption algorithm and sends back its **Digital Certificate** (which includes its Public Key). 
- **Verification:** Your browser verifies the certificate against built-in trusted authorities. If it is valid, the browser trusts the server. 
- **Key Exchange:** The browser generates a random, temporary secret (called a _Pre-Master Secret_). It encrypts this secret using the server's **Public Key** and sends it over. 
- **Session Keys Created:** Because only the server has the matching **Private Key**, only the server can decrypt it. Now, both sides use that shared secret to calculate identical **Session Keys**. 
- **Finished:** Both sides send an encrypted "Finished" message. From this point forward, every single piece of data is completely scrambled using the Session Key.

## How Certificates are Signed and Verified?

To prevent a hacker from pretending to be `google.com`, websites use **SSL/TLS Certificates** issued by trusted third parties called **Certificate Authorities (CAs)** like [Let's Encrypt](https://letsencrypt.org/) or DigiCert.

```
[ Website Owner ] -> Generates Public/Private Key -> Sends Public Key to CA 
[ CA Authority ] -> Verifies ownership -> Signs Certificate with CA Private Key [ Your Browser ] -> Verifies signature using built-in CA Public Key (Trust)
```

- **The Lock & Key Setup:** The website owner generates a math pair: a **Public Key** (given to everyone) and a **Private Key** (kept top secret). 
  
- **The Request:** The owner sends their Public Key and business details to a CA. 
  
- **The Signing:** The CA verifies that the owner actually owns the domain. The CA then takes the website's info, runs it through a math formula to create a unique fingerprint (hash), and encrypts that fingerprint using the **CA's own Private Key**. This encrypted fingerprint is the **Digital Signature**. 
  
- **The Browser's Trust:** Your operating system and browser come pre-installed with the Public Keys of all major CAs. When you visit a site, your browser decrypts the signature using the CA's public key. If the math matches, the browser knows the certificate is authentic and untouched.

## How the Encryption Works (The Transition)

Websites use a clever combination of **two different types** of encryption to balance maximum security with high speed. 

| Encryption Type             | How it Works                                                                               | Pros & Cons                                               | When it is used                                                                          |
| --------------------------- | ------------------------------------------------------------------------------------------ | --------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **Asymmetric** _(Two Keys)_ | Uses a public key to lock data, but requires a totally different private key to unlock it. | Highly secure but very slow and computationally heavy.    | Used **only** during the TLS Handshake to safely pass the initial secret.                |
| **Symmetric** _(One Key)_   | Uses the exact same single key (Session Key) to both lock and unlock the data.             | Incredibly fast and lightweight for computers to process. | Used for **everything else** (loading text, images, videos) after the handshake is done. |

Once the handshake is completed, the heavy, slow asymmetric keys are put away. The browser and server use their identical, lightweight **Session Keys** to instantly lock and unlock every packet of data traveling across the internet.
