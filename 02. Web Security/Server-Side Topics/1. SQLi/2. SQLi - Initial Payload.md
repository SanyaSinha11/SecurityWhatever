For any user controlled input field, we should usually start with low-impact detection payloads.

1. Single quote
   ```sql
   '
   ```
   Can trigger a SQL syntax error if input is embedded in a query.
   
   Usually gives:
   - Error messages
   - HTTP 500 responses
   - Different application behavior
   
2. Double quote
   ```sql
   "
   ```
   Some applications use double quotes in queries or backend processing. Works similar to single quotes. 
   
3. Boolean tests
   Submit a normal value and compare responses.
   ```sql
   True Condition
   ' AND '1'='1
   
   False Condition
   ' AND '1'='2
   ```
   
   Check for:
   - Login behavior changes
   - Different error messages
   - Different content lengths
   - Different response times
   
4. Numeric context texting
   If parameter appears numeric
   ```sql
   Normal:
   1
   
   Test for:
   1'
   OR
   1 AND 1=1
   Compare against:
   1 AND 1=2
   ```  
   
5. Parenthesis testing
   ```sql
   ')
   or
   "))
   ```
   Useful when input is wrapped inside nested queries.
   
6. Comment character testing
   Depending on database
   ```sql
   '--
   '#
   '/*
   ```
   These can help identify how the backend query is structured.
   
7. Time based detection (Authorized test only)
   If responses are identical and no errors are shown, testers often check whether database functions execute by observing response timing.
   
   The exact functions differ by database platform, but the general principle is:
   ```sql
   Condition TRUE → response delayed
   Condition FALSE → normal response
   ```
   A measurable delay can indicate SQL execution even when no visible output exists.

## What to Compare?

Don't focus only on visible content. In Burp Repeater compare:
- HTTP status code
- Response length
- Response body
- Redirect behavior
- Presence/absence of error messages
- Response timing

Many SQLi findings are discovered through subtle response differences rather than obvious database errors.
