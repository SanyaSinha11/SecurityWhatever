# **Redirects**

### Overview

An open redirect occurs when a web application accepts a user-controllable input (like a URL parameter) and redirects the user to that destination without validating if the address is safe.
Because the link begins with a trusted, legitimate domain, victims often let their guard down, making it highly effective for social engineering. 

### Types of Redirects

- **Server-Side Redirects -** 
  The most common form, where the application sets the Location HTTP response header based on unsanitized user input.

- **Client-Side (DOM-based) Redirects -** 
  Occur when JavaScript on the page takes a URL from a source (like the URL fragment or a query parameter) and passes it to a "sink" that triggers navigation, such as window.location.
  
  
- Redirect = “Server telling browser where to go next.”
- Open Redirect = “Server lets attacker choose where the browser goes next.”
- Secure auth systems = “Only allow pre-approved, trusted return addresses.”
  
### How to perform Redirects

- **Root-Relative Path (eg., `/bing.com`)** - Go to the same site, starting from the root, to `/bing.com`. So the browser resolves the path to http://example.com/bing.com, that would be just the path on the same domain.
  
- **Protocol/Scheme-Relative URL (eg., `//bing.com`)** - Keep the current protocol (http or https), but go to host bing.com. That means if you are https://example.com it would go to https://bing.com, if you are on http://example.com it would go to http://bing.com. This does redirects, commonly used in CDNs, External resources and sometimes abused in open redirect `/`injection attacks.
  
- **Weird but absolute-path Reference (eg., `///bing.com`)** - This one is strange but the browser parses it after normalizing http:///bing.com to http://bing.com and treating it effectively. In URL syntax, after `https: //host = authority (domain)`, extra `/` are usually collapsed `/ `normalized by the browser. That’s why attackers sometimes use `///evil.com` — it can look like a path but behave like an external redirect.

#### Example
- https://gm-stage.tekion.xyz/%2f%2fintigriti.com/ > decodes to > https://gm-stage.tekion.xyz///intigriti.com/ > requesting a path on the same host, not telling the browser to navigate to intigriti.com. 
- If some app blocks strings like `//evil.com`, we can use `%2f%2fevil.com` as app might not match that pattern, but after decoding it’s equivalent.
- If the href resolves to a string starting with `//host`, browsers take it as protocol-relative external link. But if the `//` is in the path part after the host (even via encoding), it stays internal.

### Mitigation & Prevention

- **Avoid Redirects:** The most effective defense is to remove user-controllable redirection functionality entirely.
  
- **Use Relative Paths:** Force redirects to be relative (e.g., `/dashboard` instead of `https://site.com`) and validate that they start with a single `/` to prevent protocol-relative bypasses like `//evil.com`.****
  
- I**mplement Allow-Lists:** Maintain a strict list of allowed destination URLs or domains and reject any input that does not match.
  
- **Indirect References:** Instead of passing a full URL, use an ID or index that maps to a hardcoded URL on the server.
  
- **User Notification:** If external redirects are necessary, implement an interstitial page warning users they are leaving the site.
