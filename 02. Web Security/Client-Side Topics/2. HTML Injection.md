# **HTML Injection**

### Overview

**HTML Injection** is a web security vulnerability where an application improperly handles user input, allowing an attacker to insert malicious HTML code into a webpage. When victims view the page, the browser interprets the injected code as part of the legitimate website, leading to phishing, UI defacement, or data theft.

### ... via Parameter Issue

Input attribute breaking trick - `“><a href=“///bing.com”>Click</a>`

- here, `“>` closes the existing HTML attribute value and the tag, making the anchor tag work as a real DOM.

The payload closes the existing tag, breaks out of the attribute, and injects a new `<a>` tag that the browser treats as real HTML.

  
**For Security:** Never insert user input directly into HTML (like `<div>USER_INPUT</div><input value="USER_INPUT”>`) instead use:
- HTML-escape it (&lt;, &gt;, &quot;, etc.)
- Or use safe DOM APIs (e.g., textContent, not innerHTML)

### Primary Types

1. **Stored (Persistent) HTML Injection:** 
   The malicious HTML code is saved directly to the application's database. Whenever other users visit the compromised page (e.g., a forum post), the payload executes automatically.

2. **Reflected (Non-Persistent) HTML Injection:** 
   The injected code is embedded into a URL parameter or an HTTP request. The attacker must use social engineering to trick victims into clicking a crafted link for the payload to execute.
   
3. **DOM-Based HTML Injection:** 
   The vulnerability exists in the client-side JavaScript itself. The attacker modifies the Document Object Model (DOM) to trick the browser into executing the injected HTML.

### Prevention and Mitigation

Securing an application requires strict adherence to secure coding practices:
1. **Output Encoding:** Before rendering user input into the browser, convert all special characters into their HTML entity equivalents (e.g., `<` becomes `&lt;` and `>` becomes `&gt;`). Most modern frameworks (like React, Angular, or Vue) handle this automatically.
   
2. **Input Sanitization:** Validate that all user-supplied data fits the expected format. Use trusted security libraries (such as DOMPurify for JavaScript) to scrub user-generated content of dangerous HTML tags and attributes.
   
3. **Content Security Policy (CSP):** Implement a robust CSP HTTP header to restrict where scripts and content can be loaded from, mitigating the overall impact if an injection succeeds.
   
4. **Avoid Unsafe DOM Methods:** Do not use unsafe JavaScript methods like `innerHTML` or `document.write()` without proper sanitization
