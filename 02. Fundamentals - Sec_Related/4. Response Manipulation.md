# Overview
Response manipulation is a security testing technique where an attacker or penetration tester intercepts and alters data traveling from a **server back to a client (browser or mobile app)**.

Unlike standard web application attacks that target the server (like SQL Injection), response manipulation targets the **client-side environment**. It exploits a fundamental flaw: **Client-side controls cannot be trusted.** If an application relies entirely on the browser to enforce security rules, a tester can change the server's response to bypass those rules completely

# When to Perform?
We perform response manipulation during the **vulnerability assessment and exploitation** phases of a penetration test. It is highly effective when we suspect an application handles security logic on the front-end rather than validating it on the back-end.

Common indicators that an application is vulnerable include:
- **Hiding UI Elements:** A menu or button is hidden based on a user role parameter in a JSON response (e.g., `"isAdmin": false`).
- **Client-Side Validations:** A form uses JavaScript to block certain inputs or characters, but the server doesn't re-check them.
- **Multi-Step Workflows:** Steps in a process (like a checkout or password reset) rely on the browser receiving a specific success message to advance to the next screen.

# Where to Perform?
Response manipulation takes place **in-transit (man-in-the-middle)** between the web server and the client browser. 

Because modern web traffic is encrypted via HTTPS, you cannot easily modify this data on the local network wire. Instead, you must perform it inside a **Local Intercepting Proxy** (like Burp Suite or OWASP ZAP) installed on your testing machine. The proxy terminates the SSL/TLS connection from the browser, decrypts the traffic, allows you to edit it in plain text, re-encrypts it, and passes it along.

# How to Perform?
Here are three real-world scenarios showing exactly how response manipulation works in practice.
#### Scenario 1: Bypassing Role-Based Access (Privilege Escalation)
- **The Setup:** You log into a web app as a regular user. The server sends a JSON profile response to your browser.
  
- **The Manipulation:** You intercept the response and look for privilege flags.
    - _Original:_ `{"username": "john_doe", "role": "user", "canDeleteUsers": false}`
    - _Modified:_ `{"username": "john_doe", "role": "admin", "canDeleteUsers": true}`
       
- **The Result:** The browser processes the modified JSON, unlocks the admin dashboard, and reveals hidden administrative buttons.

#### Scenario 2: Two-Factor Authentication (2FA / OTP) Bypass
- **The Setup:** You trigger an OTP code to your phone. You type in a random, incorrect 6-digit code (`111111`) to see how the application behaves.
  
- **The Manipulation:** The server checks the code, sees it is wrong, and sends back an error response. You intercept this response before it hits your browser.
    - _Original Response:_ `HTTP/1.1 401 Unauthorized \n {"status": "error", "message": "Invalid OTP Code"}`
    - _Modified Response:_ `HTTP/1.1 200 OK \n {"status": "success", "message": "OTP Verified"}`
      
- **The Result:** The browser is tricked into believing the server accepted the code. JavaScript redirects you past the 2FA screen straight into the account.

#### Scenario 3: Client-Side Feature Unlocking (Paywall Bypass)
- **The Setup:** You visit a premium news site or software portal. The server checks your subscription status.
  
- **The Manipulation:** You intercept the subscription check response.
    - _Original:_ `{"subscriptionActive": false, "daysRemaining": 0, "tier": "free"}`
    - _Modified:_ `{"subscriptionActive": true, "daysRemaining": 365, "tier": "premium"}`
      
- **The Result:** The browser unblocks the premium content or enables restricted premium features locally.

# Note: The "Follow-up" Requests Problem

Response manipulation is excellent for finding flaws, but it often reveals a secondary constraint.

If you manipulate an account balance response from `$0` to `$1,000,000`, the browser UI will display one million dollars. However, if you try to _spend_ that money, the browser sends a new request to the server. If the server evaluates your true balance on the back-end during the purchase request, your transaction will fail.

Response manipulation only leads to a **true critical vulnerability** if the server blindly trusts subsequent actions dictated by the manipulated client state.

# Advanced Use-Cases
Response manipulation is critical for testing vulnerabilities that cannot be found by simply changing requests.

- **Bypassing Client-Side WAFs:** 
  Some Web Application Firewalls or security scripts are delivered via JavaScript. By stripping out the security script from the HTML response, you can completely disable the protection for your session.
  
- **Testing Client-Side Protections (XSS Filters):** 
  You can inject test payloads into a server's response to see if the browser's rendering engine or local scripts sanitize data correctly, helping you find Document Object Model (DOM) Cross-Site Scripting.
  
- **API Insecure Direct Object References (IDOR):** 
  When an API sends too much data in a response (e.g., sending an entire user object including hashed passwords, even though the UI only displays the username), you manipulate the response to see if the client-side code exposes hidden parameters.

# Common Roadblocks and How to Fix Them
When manipulating responses, you will frequently encounter protections that break your connection. Here is how to bypass them:

#### Challenge A: Transport Layer Security (TLS) Pinning
- **The Problem:** Mobile apps often use TLS pinning, meaning they will reject Burp Suite’s certificate and refuse to process the manipulated response.
  
- **The Fix:** You must root/jailbreak the device and use tools like **Frida** or **Objection** to bypass the certificate pinning in memory.

#### Challenge B: Digital Signatures and Hashes
- **The Problem:** The server sends a response along with a cryptographic signature header (like `X-Signature: hmac_sha256(...)`). If you alter the response body, the signature will no longer match, and the app will throw an error. 
  
- **The Fix:** You must reverse-engineer the client JavaScript to find the signing key, or use a Burp Suite extension (like _Crypto Auditor_ or custom Python scripts) to automatically recalculate and replace the signature header when you edit the body.

#### Challenge C: Content Security Policy (CSP)
- **The Problem:** A strict CSP header sent by the server may stop you from executing scripts, even if you successfully inject them into the HTML response.
  
- **The Fix:** Use Burp's **Match and Replace** rules to search for the `Content-Security-Policy` header in responses and completely delete it or replace it with a blank value.
