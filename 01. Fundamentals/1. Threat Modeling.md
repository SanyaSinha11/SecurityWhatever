# Overview
Threat Modeling is a structured way to identify and prioritize security risks in a system before or during its development. It is a core part of SDLC.

Threat modeling is a structured process used to identify, evaluate, and mitigate potential security risks in an application or system. Think of it as home security planning: you look at your house, identify weak points like a back window, figure out how a burglar might get in, and install a lock before a break-in happens.

## Why do we do? 
- If we do't know what we have to protect then how can we protect?
	*TM answers this question. The goal is to understand what  are we protecting and whom against we are protecting , what could go wrong, how likely or impactful it is, what mitigation  to  implement*
	
Organizations perform threat modeling to build security into their products from the very beginning rather than trying to fix flaws later.
- **Reduces Costs**: Fixing a design flaw during the planning phase is significantly cheaper than fixing a breach after release.
- **Prioritizes Defense**: It helps security teams focus their limited time and budget on the most critical vulnerabilities.
- **Drives Security Awareness**: It forces developers and product owners to think like attackers, improving the overall security culture.

## When it should be performed?
Before Development after PRD
- Whenever there is a new feature 
- Whenever there are changes in the architecture
Overall its a continuous process.

Threat modeling should begin during the **earliest design phases** of a project, before a single line of code is written. However, it is not a one-time event. You must revisit and update your threat model whenever:
- New features or major code updates are added.
- System architecture or data flows change.
- The system is integrated with a new third-party service.

	1.  Scope Definition : Finding all the assets

		- Identifying all the entry points in the system where a potential attacker  could interact with a system 
		- Understand the system from stories. Stories tell about the features and implementation with that we can also infer what is not normal for that feature 

→ After this step We have all the assets we want to protect, We know all the entry points in our system where a potential attacker could interact. Also we know  trust boundaries in our system.

Now we know what to protect the next thing is to find out what are the threats we are protecting our assets against: 
	1.  Determine Threats : In order to determine threats, there are a number of threat modeling framework such as :
		 - STRIDE 
		 - DREAD 
		 - MITRE ATT&CK 
	   Using any of these framework we can determine what  could be the possible threats for our assets and how our trust boundaries can be violated?
	   Objective :  Identify → Categorize  → Find Likelyhood & Impact Matrix → Risk Matrix

In simpler terms:
- **What are we building?** Create a Data Flow Diagram (DFD) to map out user inputs, data storage, and external connections.
  
- **What can go wrong?** Brainstorm potential attacks, data leaks, or system failures using a threat model framework.
  
- **What are we going to do about it?** Decide on countermeasures, such as adding encryption, firewalls, or stronger login checks.
  
- **Did we do a good job?** Review the implemented fixes to ensure they successfully blocked the identified threats.

# Integrating Threat Modeling into Secure SDLC
To build a Secure Software Development Lifecycle (SDLC), threat modeling must be treated as a continuous activity rather than a gatekeeping checkpoint.

```bash
[Requirements]
     |
     ▼
[Design / Architecture]  ──> *PRIMARY THREAT MODELING ZONE*
     |
     ▼
[Development] 
     |
     ▼
[Testing]    ──> *VERIFICATION*
     |
     ▼
[Deployment]      
```

#### Where Exactly to Use It:
- **The Design Phase (Primary Insertion Point)**: This is where the main threat modeling session happens. Software architects, developers, and security engineers sit down with system diagrams to map out threats and write down security requirements before coding begins. 
  
- **User Story Creation (Agile Sprint Planning)**: When product managers write a new feature ticket (e.g., "As a user, I want to upload a profile picture"), they should include a mini threat model note (e.g., "Threat: User uploads malware instead of an image. Mitigation: Validate file extensions and scan files").
  
- **The Testing Phase (Verification)**: Penetration testers and QA teams use the completed threat model document as a cheat sheet. It tells them exactly where the system's weakest points are so they can test if the defenses actually work.

# Frameworks and Models within Threat Modeling
Different methodologies help teams brainstorm what can go wrong. The most common frameworks include:

1. STRIDE - A Microsoft-created developer-focused model used to categorize threats.
2. DREAD - A risk rating system (as previously discussed) used to score and prioritize the threats found during STRIDE analysis.
3. PASTA - Process for Attack Simulation and Threat Analysis. A risk-centric, business-focused model that aligns security risks directly with business impacts.
4. VAST - Visual, Agile, and Simple Threat modeling. An automated approach designed to scale across large Agile development teams using specialized software.

## STRIDE
**STRIDE** is a checklist created by Microsoft to help developers and security teams think like hackers. Its main goal is to figure out how a system might be attacked, so they can fix the vulnerabilities _before_ the system goes live. 

- **S - Spoofing:** Pretending to be someone or something else. (Example: A hacker creating a fake login page to steal your password). 
- **T - Tampering:** Unfairly modifying data or files. (Example: A hacker secretly changing the price of an item in a shopping cart). 
- **R - Repudiation:** Denying that an action ever happened. (Example: A user deleting transaction logs to claim they never bought an item). 
- **I - Information Disclosure:** Leaking private or secret data. (Example: A hacker stealing personal customer details or credit card numbers). 
- **D - Denial of Service (DoS):** Shutting down a service so real users can't access it. (Example: Flooding a website with so much fake traffic that it crashes). 
- **E - Elevation of Privilege:** Upgrading your own account to gain unauthorized power. (Example: A regular user hacking the system to become an administrator).

## DREAD
The **DREAD** framework is a scoring system used by cybersecurity teams to measure how severe and dangerous a potential computer security threat is, helping them decide which problems to fix first.

- **D** - **Damage potential**: How much harm could this threat cause if a hacker succeeds?
- **R** - **Reproducibility**: How easy is it for the attacker to repeat the hack over and over?
- **E** - **Exploitability**: How much skill or effort does it take for someone to actually launch the attack?
- **A** - **Affected users**: How many people or systems would be impacted if it happens?
- **D** - **Discoverability**: How easy is it for hackers to find this specific weak spot in the first place?
